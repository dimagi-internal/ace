# Migration: 0.13.344 — decisions schema v2

**Date:** 2026-05-22

## What changed

- `decisions.yaml` schema bumped from version 1 to version 2.
- Per-row field rename: `default:` → `ai-default:`.
- New optional per-row field: `override:`. Set when a human overrides
  via the renderer + sync flow. Effective value = `override` if present
  else `ai-default`.
- `status` enum changed: `open` removed. Old `open` rows collapse into
  `applied` (the orchestrator no longer blocks on `open` rows; the
  distinction was load-bearing only at gate-brief authoring time, which
  was removed in 0.13.116).

See `docs/superpowers/specs/2026-05-22-fork-decisions-modes-design.md`
for the full design.

## Action required

**None for in-flight runs.** `parseDecisionsYaml` transparently upgrades
v1-shape input to v2 in memory; the next write (via `decisions-render`
or `decisions-sync`) persists the v2 shape to disk. Existing
`decisions.yaml` files on Drive in v1 shape continue to read correctly
until they're rewritten naturally.

A consequence: a `decisions.yaml` may stay in v1 shape on disk
indefinitely if nothing rewrites it. That's fine — readers handle both
shapes — but the file diff against the new fixtures will look
mismatched until a rewrite happens.

## Rewriting v1 files in bulk (optional)

If you want to proactively upgrade all `decisions.yaml` files on Drive
to v2:

```bash
for opp in $(list-of-opp-slugs); do
  for run in $(list-of-runs-under-opp); do
    /ace:step decisions-render <opp>/<run>
  done
done
```

`decisions-render` reads the YAML (upgrading v1 to v2 in memory),
re-renders the gdoc, and triggers a YAML rewrite at the next sync. Or
trigger the rewrite directly via `decisions-sync` (no-op if the gdoc
hasn't been edited; still re-serializes the YAML).

## Rollback

This migration is one-way at the in-memory level. v2-shape
`decisions.yaml` files on disk cannot be parsed by ACE plugin <0.13.344.
Downgrade requires restoring affected files from a Drive snapshot. No
schema-v2 → v1 converter is provided — the conversion would have to
guess where to put the `override:` value (a v1 row has no slot for it
without destroying the AI-default).
