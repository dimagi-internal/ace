# CommCare 2.63.0 backup-code entry UX

**Date:** 2026-05-22
**Connect APK release:** `commcare_2.63.0` (published 2026-05-20)
**PR landing:** PR #415 (plumbing) + follow-up flip-default PR

## What changed in 2.63.0

Two breaking changes to the registration flow:

1. **Release-asset filename.** GitHub release `commcare_2.63.0` ships the APK as `commcare-2.63.0-release.apk` (version embedded in filename) instead of the legacy `app-commcare-release.apk`. The MCP downloader at `mcp/mobile/client.ts:675` now probes the new filename first and falls back to the legacy name on 404 — keeps pre-2.63.0 pins working without code change.

2. **Backup-code entry UX.** The single `id/connect_backup_code_input` text field (plus the once-existed-then-removed `id/connect_backup_code_repeat_input`) was replaced with a six-cell PIN-style widget. The new layout:
   - `id/backup_code_view` (LinearLayout) contains six unidentified single-char `EditText` children
   - Maestro's `inputText` **auto-advances** across the cells — no per-cell tap needed
   - Initial-entry + re-confirmation collapsed into one combined "Confirm Backup Code" screen (welcome_back header visible from first paint)
   - CONTINUE button retains its old id (`id/connect_backup_code_button`)
   - New companion elements: `id/recovery_code_tilte` (sic — typo in upstream), `id/backup_code_subtitle`, `id/backup_code_visibility_toggle`

## Recipe-level fix

`mcp/mobile/recipes/static/connect-register-from-otp.yaml` now branches on which gating selector is visible:

```yaml
# 2.63.0+ path: combined "Welcome Back + Backup Code" with six-cell PIN widget.
- runFlow:
    when:
      visible:
        id: "org.commcare.dalvik:id/backup_code_view"
    commands:
      - tapOn:
          id: "org.commcare.dalvik:id/backup_code_view"
      - inputText: ${BACKUP_CODE}
      - waitForAnimationToEnd:
          timeout: 2000
      - tapOn:
          id: "org.commcare.dalvik:id/connect_backup_code_button"

# 2.62.0 path: single-field initial entry (fallback for explicit version pins).
- runFlow:
    when:
      visible:
        id: "org.commcare.dalvik:id/connect_backup_code_input"
    commands:
      - tapOn:
          id: "org.commcare.dalvik:id/connect_backup_code_input"
      - eraseText: 12
      - inputText: ${BACKUP_CODE}
      - hideKeyboard
      - tapOn:
          id: "org.commcare.dalvik:id/connect_backup_code_button"
```

The two `runFlow.when` branches both skip cleanly when their gating id isn't visible, so a single recipe handles both Connect versions without runtime branching.

## What stayed the same

Verified end-to-end via `mobile_register_test_user` on a fresh 2.63.0 AVD:

- All phone-entry, OTP-bypass, PIN-entry, Android-system-PIN-setup, lockscreen, and name-entry selectors carried over unchanged from 2.62.0 (~14 selectors).
- "Account Recovered" success dialog (`connect_message_title`, `connect_message_button`) unchanged.
- CommCare home screen post-registration unchanged: `screen_first_start_main`, `str_setup_message` ("Welcome to CommCare!"), `connect_login_button` ("GO TO CONNECT MENU"), `enter_app_location` ("Enter Code"), `btn_fetch_uri` ("Scan Application Barcode").
- Nav drawer items (`parent_item_container` + `list_title` for Opportunities / CommCare Apps / Messaging / Work History) unchanged.
- Bottom-bar items (`notification_view`, `about_view`, `app_version` confirms `v 2.63.0`) unchanged.

## What's not yet verified at 2.63.0

The bootstrap path through home screen is solid. Phase 6 inner-loop surfaces — opportunity tile claim, Learn module navigation, Deliver form submission, sync flow — are reachable via standard nav-drawer selectors but haven't been exercised end-to-end against 2.63.0. Any drift there will surface in the first live Phase 6 dispatch and patches go in `connect-2.63.0.yaml` and the relevant recipe.

## Discovery notes

Initial drift symptom: `mobile_register_test_user` failed on `Assert that id: org.commcare.dalvik:id/connect_backup_code_input is visible` at the `extendedWaitUntil` step. Captured UI dump showed `id/backup_code_view` in its place. Manual hypothesis test (`adb shell input tap <first-cell>` + `adb shell input text 222222`) confirmed the widget auto-advances through all six cells — turning what looked like a six-call recipe rewrite into a one-line gating-selector swap. The whole drift class was easier than feared.

The lesson, again: when an Android OS-level widget surface changes, hypothesis-test the input semantics via direct ADB **before** writing recipe loops — the platform often does the auto-advance / auto-mask / focus-chaining work for you.
