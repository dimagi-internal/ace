# CI-660 — `start_learn_app` 500 + Silent Android-Client Noop (Phase 5)

**Status:** Open upstream as [CI-660](https://dimagi.atlassian.net/browse/CI-660). Mitigated client-side in ACE by `connect_preflight_learn_app_user` boundary probe (this PR).

**Origin:** PR #249 (commit `8677225`, 2026-05-12) — Phase 5 mobile recipe stalled mid-navigation because the opp-detail "Start" button appeared to noop. Network capture revealed `POST /users/start_learn_app/` returning Django HTTP 500 with no JSON body; the Connect Android client logs the failed response and swallows it.

## What looked like the bug (prior framing)

> "Phase 5 mobile recipe stalled at the 'Start learning' button — selector
> drift? Maestro driver wedge? AVD cookie expiry?"

We chased four AVD-side hypotheses before realising the server was 500ing every claim. The classic shape of a silent client noop: the *client* looked broken; the *server* was the actual breakpoint, masked by the client's defensive-swallow.

## What the bug actually is

`commcare-connect/users/views.py:107` (`start_learn_app` view):

```python
def start_learn_app(request, opportunity_id):
    ...
    user_link = create_hq_user_and_link(user, opportunity, ...)  # <-- not in try/except
    ...
```

`create_hq_user_and_link()` issues a `POST /a/<domain>/api/v0.5/user/` against CCHQ to mint (or re-link) the mobile worker. Any CCHQ-side rejection — auth failure, missing `connect_username` permission (COMMCARE_CONNECT flag off), domain archived, user already linked to a different ConnectID — raises `CommCareHQAPIException`. Because the view doesn't wrap the call, the exception bubbles all the way out and Django returns HTTP 500 with a generic error page.

The Connect Android client's response handler logs the failure (Sentry shape doesn't match a structured JSON error) and swallows it, so the FLW sees the "Start" button click do nothing.

## Sibling: same class as the `short_description` 50-char trap

Identical structural pattern to the `short_description` trap documented in `docs/learnings/2026-05-12-connect-opp-short-description-50-char-trap.md`:

|  | `short_description` 50-char trap | CI-660 `start_learn_app` |
|---|---|---|
| Symptom | HTTP 500, no actionable body | HTTP 500, no actionable body |
| Server-side cause | Postgres `DataError` bubbles past narrow `except` clause in `program/api/views.py:102` | `CommCareHQAPIException` bubbles past missing try/except in `users/views.py:107` |
| Client-side symptom | "Phase 3 mysteriously 500s" | "Phase 5 button silently noops" |
| ACE-side mitigation | Zod `.max(50)` cap on the atom input | `connect_preflight_learn_app_user` boundary probe |
| Upstream fix | Align serializer to model (or vice versa) | Wrap `create_hq_user_and_link` in try/except, return structured JSON error |

The boundary-probe-registry pattern (`docs/learnings/2026-05-12-boundary-probe-registry.md`) is the umbrella for both. CI-660 is the second row to graduate from **Pending** to **Shipped** in 12 hours.

## What the shim does

`connect_preflight_learn_app_user(hq_domain, connect_username?, api_key, hq_username, base_url?)` runs two cheap CCHQ probes and returns a structured `{ok, action, reason}` outcome:

1. **`GET /a/<domain>/api/v0.5/user/?limit=1`** — same call the doctor's `hq_api_key_auth` probe runs (`bin/ace-doctor:1420`). Validates that the HQ API key works against the requested domain. Catches:
   - 401/403 → `auth_failed` (rotate-key remediation in the error message).
   - 404 → `domain_unreachable` (typo / archived).
   - 5xx → `cchq_error` (CCHQ outage; defer the run).
   - network throw → `domain_unreachable` (connectivity).
   - 2xx → continue.
2. **`GET /a/<domain>/api/v0.5/user/?username=<connect_username>`** (only when the caller supplied `connect_username`) — checks whether the mobile worker `create_hq_user_and_link` is about to mint already exists. Three branches:
   - Empty list → `would_create`. Happy path.
   - User found, `is_active=true`, `connect_username` matches (or empty) → `would_reuse_existing`. Happy path; the upstream `get_or_create`-style link helper is idempotent for this case.
   - User found, but `connect_username` mismatch or `is_active=false` → `conflict_existing_user`. Halts Phase 5 with a clear remediation: pick a different username, or manually reconcile the HQ record.

When `connect_username` is omitted (the common case — ACE doesn't always know the FLW's ConnectID at opp-create time), only step 1 runs. That alone catches the most common CI-660 failure modes (rotated API key, archived HQ domain, CCHQ outage) at near-zero cost.

The probe is read-only — it never mutates CCHQ state. Safe to re-run; safe to wire as a doctor probe in the future if we want it to run unconditionally at session start.

## Design choice: why a new atom, not a check inside `connect_send_flw_invite`

`connect_send_flw_invite` POSTs to Connect's `invite_users/` endpoint, which is a Connect-side operation (it queues a `UserInvite` row + an SMS). It does *not* go through CCHQ — the CCHQ user-create happens later, asynchronously, when the FLW actually claims the opp from the Android client. Inlining a CCHQ pre-flight there would be a misnomer.

The shim is a Phase-5-prerequisite, not a Phase-3-prerequisite. It lives best as its own atom that the producer skill (`connect-opp-setup` Step 7.5) calls explicitly. The pattern matches how `ocs_shared_collection_team` is a standalone doctor probe rather than a hidden check inside `ocs_clone_chatbot`.

## Upstream (commcare-connect)

Wrap `create_hq_user_and_link` in try/except inside `users/views.py:107` (and any other consumer of `CommCareHQAPIException` that returns a generic 500). Return a structured JSON error so the Android client can surface a real banner to the FLW instead of swallowing the failure. CI-660 tracks this.

Once upstream lands, the ACE-side shim stays as defense-in-depth — auth/domain failures are still cheaper to catch pre-Phase-5 than at the end of an AVD navigation sequence.

## Pattern characteristics (for the registry)

This probe matches the five characteristics from the **Boundary Probe Registry**:

- **~250ms cost order.** Two GETs against CCHQ. Same endpoint family the doctor already calls.
- **Catches a class-of-bug.** Any CI-660-like failure (auth, domain, user-conflict) — not just the one we discovered in PR #249.
- **Names the remediation in the error.** The `reason` field embeds the literal rotate-key / reactivate-user / fix-domain command.
- **Right boundary.** Phase 5 prerequisite, attached to the producer skill (`connect-opp-setup`) that creates the opp this user will claim.
- **Idempotent + safe to retry.** Read-only; never mutates.
