# Nova `validate_app` passes apps with unescaped XML metacharacters in field text

**Filed:** 2026-04-30 from ACE turmeric-20260429-2330 e2e run
**Affects:** voidcraft-labs/nova-plugin (latest as of 2026-04-30; tested with `add_fields` and `edit_field`)
**Severity:** P2 — silent until CCHQ build, then deterministic build failure with a misleading error

## Summary

Nova's `validate_app` returns `{"success": true}` for apps whose field
`label`, `hint`, or option text contains literal `<`, `>`, or `&`
characters that are not encoded as XML entities. Since XForm XML is
Nova's own output format, an unescaped `<` in a field label produces an
XForm document that is not well-formed XML and that **cannot be released
on CommCare HQ**: HQ's build endpoint returns

```
Error on line 119 column 64: not well-formed (invalid token)
```

…where line 119 is somewhere deep inside the auto-generated XForm. The
operator sees a build error referencing a line in machine-generated XML,
not the original field they typed.

## Reproducer

PDD field text generated by the architect on the turmeric e2e:

```
Q10 — Stock quantity (<2kg, 2-10kg, >10kg)
```

The architect called `add_fields` / `edit_field` with the literal label
`Q10 — Stock quantity (<2kg, 2-10kg, >10kg)`. Nova accepted the write,
the field round-tripped through `get_form` with the unescaped chars, and
`validate_app` on the containing app reported `{"success": true}`.

`upload_to_hq` succeeded (Nova's draft format is JSON, not XML; the JSON
encoder escapes nothing because it doesn't have to). The CCHQ-side
"build a release" step is where the JSON gets rendered as XForm XML,
and that's where the build fails.

## Expected behavior

Pick one of these two — either is sufficient:

1. **Auto-escape on write.** `add_field` / `add_fields` / `edit_field`
   XML-escape `label`, `hint`, and option-label text on the way in, the
   same way they do for any other content destined for the XForm body.
   The user types `(<2kg)` and the stored value is `(&lt;2kg)`.
2. **Reject in `validate_app`.** `validate_app` walks every form's text
   and returns `{"success": false, errors: [{ field: "q10", reason: "label contains unescaped '<' — would fail XForm well-formedness" }]}`.

Auto-escape is the better fix — XML escaping is a pure-output concern,
authors shouldn't have to know it. `validate_app` already runs late, so
the error message would still surface only at app-build time, but it
would surface in Nova's own output instead of CCHQ's misleading line/
column message.

## Why ACE is filing this now

The ACE plugin has added a Phase 2 pre-upload XML-escape lint as a
class-level preventer (`skills/app-deploy/SKILL.md § 2.5`). That lint
fixes the symptom, but every Nova client that writes apps with arbitrary
text in labels (i.e. all of them) has the same exposure. Pushing the fix
upstream removes the lint from ACE — and from every other downstream
consumer that doesn't know to add it.

## Concrete diff in ACE that would go away

If Nova auto-escapes, ACE drops:

- `skills/app-deploy/SKILL.md § 2.5` (the lint procedure)
- the corresponding edit_field calls during deploy
- the gate-brief `[WARN]` for skipped lint
