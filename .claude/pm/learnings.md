## Preferences

- **Runtime-health checks should be WARN, not FAIL.** When adding `/ace:doctor` (or similar) checks that test whether a user is ready to *use* a feature (vs. whether the install works), use WARN. A user doing a design-review-only run shouldn't see FAIL for missing OCS config. FAIL is reserved for "tool won't run at all." (2026-04-15, from P3 implementation)
- **Skills that read external-human inputs must fail loudly, not improvise.** If a skill's input is `producedBy: 'external'` in the artifact manifest and the file isn't there, the skill must stop with an actionable error — never invent content. The orchestrator is responsible for capturing the input; the skill is responsible for defending against the bypass path (`/ace:step`). (2026-04-15, from P1 implementation)
- **Smoke-test scripts on the current machine before committing.** Bash / shell changes to `bin/` that look right in the diff can still break on live env data (quoting, quotes, empty values, etc.). Run the script at least once locally with `--here` or equivalent before staging. Caught a quote-stripping bug on 2026-04-15 that would have shipped otherwise.
- **Passing tests can still hide fixture drift against a spec/manifest.** A green `npm test` is not the same as "the fixture matches the canonical shape declared in `lib/artifact-manifest.ts`." The 2026-04-16 scout found three gaps (stale `state.yaml`, missing `test-prompts.md`, missing Phase 4–6 artifacts) that all passed CI because the test was pinned to `upToPhase: 'connect'`. When a change to the manifest / phase structure / skill contracts lands, re-check fixtures against the full manifest — don't rely on the test alone. (2026-04-16)
- **3-fixture coverage model.** `CRISPR-Test-001` = partial input fixture for `ocs-agent-setup` (Phase 1–3 inputs only); `CRISPR-Test-002` = focus-group archetype stress fixture (Phase 1–3); `CRISPR-Test-003-Turmeric` = complete E2E fixture with every required artifact stubbed across all 6 phases. When adding new skills / artifacts, update CRISPR-Test-003-Turmeric first (it's the one the manifest test validates end-to-end). (2026-04-16)
- **qa/eval two-phase pattern is canonical.** A skill that exercises an artifact at runtime (chat with a deployed bot, facilitate a group session, capture audio) should split into `-qa` (capture evidence + structural checks, no LLM) and `-eval` (judge evidence with LLM-as-Judge). Every `-eval` writes a verdict YAML to `verdicts/<skill>-<mode>.yaml` using the uniform shape in `skills/README.md § QA vs Eval`. Skills that produce their own evidence inline use a `## LLM-as-Judge Rubric` self-eval section. The umbrella `opp-eval` skill aggregates every `verdicts/*.yaml` into a run-level scorecard with recommendations — add a rubric to a skill and it auto-registers. (2026-04-19, from qa-eval-iteration-loop cycle)
- **OCS can silently block `publishChatbotVersion` on pipeline errors.** A non-existent `collection_index_id` attached via `ocs_attach_knowledge` lands without error at the pipeline-patch layer (200 OK), but then every subsequent publish call is blocked with the opaque UI message *"Unable to create a new version when the pipeline has errors."* The draft ends up correctly configured, but the widget keeps serving v1 (empty post-clone) forever. Rule: pre-flight validate collection ids exist on the team before attaching — see `scripts/bootstrap-ocs-golden-template.ts` `listCollectionIndexIds` helper (scrapes the edit page since OCS has no REST endpoint for this). Same hardening needed in `ocs-agent-setup` for every per-opp bot clone. (2026-04-19, discovered when golden template scored 3.84/10 FAIL)
- **Archetype branching is a single-skill one-PR unit of work.** Adding a `## Archetypes` section with per-archetype branches to a SKILL.md is consistently ~1 file of substantive change + CHANGELOG + version bump, tests stay green, shippable autonomously in one cycle. Two shipped this way in 2026-04-19 (pdd-to-test-prompts, llo-invite). When auditing remaining skills for silent atomic-visit defaults, each can be its own small PR — no need to batch. (2026-04-19)
- **Defensive state init on bypass paths.** `/ace:run` initializes `state.yaml` as part of "Starting a New Opportunity," but operators bypass the orchestrator via `/ace:step <skill> <opp>` or by dispatching a phase agent directly. Every entry path that touches state must tolerate a missing `state.yaml` and init first. `/ace:step` owns this for its path (see `commands/step.md` step 4); phase-agent Agent-tool dispatches are expert paths and should at minimum not crash on missing state — skip the status update with a single-line warning. Motivation: I hit this myself in the cosmetics-fgd-pilot setup when I bypassed `/ace:run` and subsequent skill dispatches had no state file to update. (2026-04-19)
