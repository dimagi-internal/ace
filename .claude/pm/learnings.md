## Preferences

- **Runtime-health checks should be WARN, not FAIL.** When adding `/ace:doctor` (or similar) checks that test whether a user is ready to *use* a feature (vs. whether the install works), use WARN. A user doing a design-review-only run shouldn't see FAIL for missing OCS config. FAIL is reserved for "tool won't run at all." (2026-04-15, from P3 implementation)
- **Skills that read external-human inputs must fail loudly, not improvise.** If a skill's input is `producedBy: 'external'` in the artifact manifest and the file isn't there, the skill must stop with an actionable error — never invent content. The orchestrator is responsible for capturing the input; the skill is responsible for defending against the bypass path (`/ace:step`). (2026-04-15, from P1 implementation)
- **Smoke-test scripts on the current machine before committing.** Bash / shell changes to `bin/` that look right in the diff can still break on live env data (quoting, quotes, empty values, etc.). Run the script at least once locally with `--here` or equivalent before staging. Caught a quote-stripping bug on 2026-04-15 that would have shipped otherwise.
- **Passing tests can still hide fixture drift against a spec/manifest.** A green `npm test` is not the same as "the fixture matches the canonical shape declared in `lib/artifact-manifest.ts`." The 2026-04-16 scout found three gaps (stale `state.yaml`, missing `test-prompts.md`, missing Phase 4–6 artifacts) that all passed CI because the test was pinned to `upToPhase: 'connect'`. When a change to the manifest / phase structure / skill contracts lands, re-check fixtures against the full manifest — don't rely on the test alone. (2026-04-16)
- **3-fixture coverage model.** `CRISPR-Test-001` = partial input fixture for `ocs-agent-setup` (Phase 1–3 inputs only); `CRISPR-Test-002` = focus-group archetype stress fixture (Phase 1–3); `CRISPR-Test-003-Turmeric` = complete E2E fixture with every required artifact stubbed across all 6 phases. When adding new skills / artifacts, update CRISPR-Test-003-Turmeric first (it's the one the manifest test validates end-to-end). (2026-04-16)
- **qa/eval two-phase pattern is canonical.** A skill that exercises an artifact at runtime (chat with a deployed bot, facilitate a group session, capture audio) should split into `-qa` (capture evidence + structural checks, no LLM) and `-eval` (judge evidence with LLM-as-Judge). Every `-eval` writes a verdict YAML to `verdicts/<skill>-<mode>.yaml` using the uniform shape in `skills/README.md § QA vs Eval`. Skills that produce their own evidence inline use a `## LLM-as-Judge Rubric` self-eval section. The umbrella `opp-eval` skill aggregates every `verdicts/*.yaml` into a run-level scorecard with recommendations — add a rubric to a skill and it auto-registers. (2026-04-19, from qa-eval-iteration-loop cycle)
- **OCS can silently block `publishChatbotVersion` on pipeline errors.** A non-existent `collection_index_id` attached via `ocs_attach_knowledge` lands without error at the pipeline-patch layer (200 OK), but then every subsequent publish call is blocked with the opaque UI message *"Unable to create a new version when the pipeline has errors."* The draft ends up correctly configured, but the widget keeps serving v1 (empty post-clone) forever. Rule: pre-flight validate collection ids exist on the team before attaching — see `scripts/bootstrap-ocs-golden-template.ts` `listCollectionIndexIds` helper (scrapes the edit page since OCS has no REST endpoint for this). Same hardening needed in `ocs-agent-setup` for every per-opp bot clone. (2026-04-19, discovered when golden template scored 3.84/10 FAIL)
- **Archetype branching is a single-skill one-PR unit of work.** Adding a `## Archetypes` section with per-archetype branches to a SKILL.md is consistently ~1 file of substantive change + CHANGELOG + version bump, tests stay green, shippable autonomously in one cycle. Two shipped this way in 2026-04-19 (pdd-to-test-prompts, llo-invite). When auditing remaining skills for silent atomic-visit defaults, each can be its own small PR — no need to batch. (2026-04-19)
- **Defensive state init on bypass paths.** `/ace:run` initializes `state.yaml` as part of "Starting a New Opportunity," but operators bypass the orchestrator via `/ace:step <skill> <opp>` or by dispatching a phase agent directly. Every entry path that touches state must tolerate a missing `state.yaml` and init first. `/ace:step` owns this for its path (see `commands/step.md` step 4); phase-agent Agent-tool dispatches are expert paths and should at minimum not crash on missing state — skip the status update with a single-line warning. Motivation: I hit this myself in the cosmetics-fgd-pilot setup when I bypassed `/ace:run` and subsequent skill dispatches had no state file to update. (2026-04-19)
- **OCS team-scoping is enforced at publish, not attach.** `ocs_attach_knowledge` with a `collection_index_id` that doesn't exist on the current team silently succeeds at the pipeline-patch layer (returns `ok: true`). Every subsequent `publishChatbotVersion` is then blocked with the opaque UI message *"Unable to create a new version when the pipeline has errors."* The real error lives on the pipeline-save response body under `errors.node.<id>.<field>`, which `/versions/create` never calls. As of 0.5.1, `publishChatbotVersion` pre-flights through `/pipelines/data/` to surface this as a typed `PipelineValidationError` with node + field — the silent-block class is closed at the MCP bottleneck. Cross-team collection references (template on team A referencing a collection on team B) hit the same rejection path; OCS does not support them. (2026-04-20, verified via Path C experiment + Iter 8 clone)
- **Pipeline-save response has two error shapes.** The OCS pipeline-save endpoint returns errors in one of two shapes depending on what failed: `{ errors: ["string1", "string2"] }` for edge-level / graph-topology errors, or `{ errors: { node: { "<node-id>": { "<field>": "<msg>" } } } }` for node-level (parameter / reference) errors. Any code that parses this response must handle both shapes — checking only the top-level string array misses all node-level errors. `extractPipelineErrors` in `mcp/ocs/backends/pipeline-patch.ts` (0.5.1) is the canonical extractor; use it rather than re-implementing. (2026-04-20)
- **Metadata files (`~/.ace/*.json`, `connect-ocs-bot.json`, etc.) are hypotheses, not truths.** The 2026-04-09 snapshot of `~/.ace/connect-ocs-bot.json` listed `shared_collection_id: 718`, which was stale by 2026-04-19. Anchoring subagent reasoning on that id propagated a wrong premise across two cycles. For any metadata file older than ~7 days that's load-bearing for the current cycle, probe the relevant fact against live state before acting. When you correct stale metadata, update the file in the same cycle to prevent re-propagation. (2026-04-20, from collection-clone-and-mcp-preflight cycle)
- **Django forms can silently-accept with missing required fields.** `add_collection_files` accepts a POST without `chunk_size` / `chunk_overlap` and returns 302 (redirect to collection home), but produces zero chunks during indexing — retrieval then silently never works. Any MCP wrapper for a Django form-backed endpoint must verify it's sending every field the form declares as required; unit tests against mock request layers miss this. (2026-04-20, from uploadCollectionFiles fix in 0.5.1)
- **Prefer class-level preventers to instance-level checks.** The first-draft fix was "add a doctor check for ACE_DRIVE_ROOT_FOLDER_ID." The shipped fix was "diff the full `.env.tpl` key set against the installed `.env`." Second draft is ~15 lines of bash for ~unlimited future coverage; it also catches the shared-collection triple and every future var automatically. Same pattern as the 0.5.1 MCP pre-flight (one bottleneck closes the whole silent-block class). Rule: when proposing a check/validation, ask "what's the class of thing this detects?" and write the check at the class level, not the instance level, whenever the class boundary is cheap to express. (2026-04-20, from env-drift-adoption-blockers cycle)
- **For adoption-blockers lens, don't trust the tool under audit as its own oracle.** Doctor reported `STATUS: COMPLETE` while the installed `.env` was missing 8 keys from `.env.tpl` — doctor only validated 3 of 16 keys. The scout almost moved on based on that green light. Read primary sources (the `.env` file, `.env.tpl`, user-facing docs) directly before trusting any "everything is green" summary from the same tooling that may itself be the gap. (2026-04-20, from env-drift-adoption-blockers cycle)
- **`.env.tpl` drifts forward across releases; installed `.env` files don't auto-update.** The canonical fix for users is `op inject -i .env.tpl -o <env-path> --account dimagi.1password.com`, which preserves existing 1Password-backed values while picking up new keys. As of 0.5.4, `/ace:doctor`'s new `env_drift` check surfaces which keys are missing from the installed `.env` with the exact `op inject` command as the fix hint. When `.env.tpl` gains a new var, no doctor-code change is needed — the diff picks it up automatically. (2026-04-20, from env-drift-adoption-blockers cycle)
- **Sibling-checkout merge discipline: verify `git branch --show-current` before any `pull/merge/push` chain.** The main checkout at `~/emdash-projects/ace/` can be left on a stale feature branch from previous work (e.g. `feat/run-smart-defaults` after PR #41 was merged and its remote deleted). A naive `cd ~/emdash-projects/ace && git pull --rebase && git merge <branch> && git push` lands the merge on that stale branch and can revive the remote branch with a fresh push. Canonical form: `cd ~/emdash-projects/ace && [ "$(git branch --show-current)" = "main" ] || git checkout main && git pull --ff-only && git merge <branch> --no-ff && git push`. Recovery if this happens: `git checkout main && git merge --ff-only <accidental-branch> && git push origin main` (works when the accidental merge commit has the real main tip as an ancestor), then delete the revived remote branch. (2026-04-20, from env-drift-adoption-blockers cycle)
- **When adding a doctor / diagnostic hint that tells the user to "run X", test that X actually runs end-to-end on the author's machine before shipping.** 0.5.4's `env_drift` check told users to run `op inject -i .env.tpl -o <env-path> --account dimagi.1password.com`. That command failed on every invocation because `.env.tpl` line 25 had an op-reference with parentheses in the item name (`ACE - OCS REST API Key (connect-ace)`) that op's parser silently truncates at. The adoption-blockers lens correctly identified the env-drift class but missed that the *hint itself was unreachable* — shipping the check without running the remediation once, left users in a worse place than no check at all. Rule: for any FAIL/WARN with a `fix:` command, run the command fresh at least once before landing. 0.5.5 fixed this by switching to a UUID-based reference. (2026-04-20, follow-up to env-drift-adoption-blockers cycle)
- **1Password vault values are hypotheses too, same as metadata files.** The 2026-04-20 earlier cycle established "metadata files older than ~7 days are hypotheses, probe against live state." The same rule applies to 1Password: `Config.shared_collection_id` still held 718 (stale, from before the 2026-04-20 clone to collection 350). A re-inject rolled the user's manually-corrected local .env value back to the stale vault value. When correcting a value in local state (e.g. `~/.ace/*.json`, `.env`), update the upstream source-of-truth in the same cycle — otherwise every inject/sync silently undoes the fix. In this case: `op item edit "ACE - Open Chat Studio" "Config.shared_collection_id[text]=350"`. (2026-04-20, from the /ace:update post-mortem)
- **op-inject references with parentheses in item names fail silently-at-parse.** `op://<vault>/ACE - OCS REST API Key (connect-ace)/credential` errors with `too few '/': secret references should have at least vault, item and field specified` — the parser truncates at space-before-paren. Percent-encoding `%28`/`%29` doesn't help (parser still bails at the literal space). Full `%20`-encoding of spaces fails differently. **Use UUID references** (`op://AI-Agents/<uuid>/credential`) for any 1Password item whose name contains parens, or rename the item in 1Password to remove them. (2026-04-20, shipped as 0.5.5)
- **Adoption-blockers scouts must run both the forward AND inverse template diff.** Forward (`.env.tpl` keys vs installed `.env`) catches keys missing from install — the 0.5.4 class. Inverse (`.env.tpl` keys vs code tree via `grep -rE "\b$KEY\b"` against `mcp/ lib/ scripts/ skills/ bin/ hooks/ agents/ commands/ test/`) catches keys declared but unread — the 0.5.7 class. Both are cheap one-liners; together they close the dead-var subclasses at the template boundary. A scout that runs only the forward diff will miss first-run ceremony that pays no runtime benefit. **Why:** 0.5.7 shipped four dead vars deletions (two visible in README step 6 as paste-this lines); each was pointless ceremony from pre-consolidation code. **How to apply:** in adoption-blockers Phase 1 Step 1, run both diffs upfront; carry any inverse-sweep hits into proposals even if the forward diff is green. (2026-04-20, from the dead-env-vars cycle, shipped as 0.5.7)
- **Before shipping a diagnostic check, exercise it against known-good inputs.** The learning from 0.5.4 follow-up ("run the `fix:` command end-to-end before landing") is the remediation side. The complementary failure is the check itself misfiring: 0.5.7's first-draft `unused_env_keys` had `grep -v '\.env'` to exclude env-file lines, which accidentally excluded `process.env.FOO` in JS/TS consumers and reported two known-good vars (OCS_USERNAME, OCS_PASSWORD) as dead. Caught because I remembered those had a consumer at `mcp/ocs-server.ts:81-82`; shipping without that mental oracle would have regressed. **Why:** a check that over-warns is worse than no check (users learn to ignore doctor). **How to apply:** for any FAIL/WARN path in `bin/ace-doctor` (or any diagnostic), run it against both states at authoring time — confirm the PASS case passes, confirm a synthetic FAIL case fails. Bash shell gotcha: `bin/ace-doctor` runs under bash (`#!/usr/bin/env bash`) but the Bash tool's default shell on macOS is zsh, which differs on unquoted-var word-splitting; explicit `bash -c '...'` is needed to reproduce doctor behavior interactively. (2026-04-20, from the dead-env-vars cycle)
- **1Password is source of truth; local `.env` is derived.** The first-run docs previously said "paste the bootstrap output into `.env`" (for OCS_GOLDEN_TEMPLATE_ID and two dead vars). But `.env.tpl` declares that var as an `op://` reference, so any future `op inject` — triggered routinely by `env_drift` WARNs when new vars land — silently reverts the pasted value to whatever 1Password holds. 0.5.7 fixed bootstrap to print `op item edit "ACE - Open Chat Studio" "Config.golden_template_id[text]=<new_id>" --vault AI-Agents --account dimagi.1password.com` followed by `op inject -i .env.tpl -o ~/.claude/plugins/data/ace-ace/.env` instead. **Rule:** never tell a user to hand-edit a key that `.env.tpl` declares as an `op://` reference — direct them to the vault. If you find a skill/doc telling users to paste into `.env`, check whether the receiving key is `op://`-backed; if so, rewrite the instruction as `op item edit` + `op inject`. (2026-04-20, shipped as 0.5.7)
- **HTTP auth probes must disable redirect-following or they false-positive on stale sessions.** OCS `isAuthenticated()` followed redirects by default, so a stale-session `302 → /login → 200` returned `true` and the auto-relogin path never fired — Phase 4 failed identically across two `/reload-plugins` cycles before the bug was isolated. Every auth liveness probe (OCS, Connect, any future session-based MCP) must use `redirect: 'manual'` (or equivalent) and assert on the *first* response — a 2xx with no redirect, or a 302 to a known logged-in route. (2026-05-02, from canopy observation 8280e1645f92)
- **macOS GUI-launched apps don't source `~/.zshrc` — `OP_SERVICE_ACCOUNT_TOKEN` and other exports are invisible.** Claude Code launched from Emdash / Dock / Finder inherits launchd's env, not the interactive shell's. `op signin` in a terminal also doesn't propagate its session token to Claude Code's bash subprocesses (fresh shells, no parent session). Fix is two-layer: a LaunchAgent that injects the token into launchd's env domain (`launchctl setenv`) **and** a `~/.zshenv` (not `~/.zshrc`) that re-acquires it in every spawned zsh subprocess. `/ace:setup` and `/ace:doctor` should detect "GUI launch + missing OP token" and surface this remediation rather than just reporting `OP_SERVICE_ACCOUNT_TOKEN: NOT SET`. Recurred across 10 canopy observations in 4 days. (2026-05-02, from canopy observations c88a51122e05, 9d68261a3309, 175abf071c61, 930a0f2bd371, dee9e7120ca1)
- **Nova OAuth has three intertwined footguns: `authenticate` is destructive, refresh tokens don't persist across MCP restarts, and `mcp-needs-auth-cache.json` makes "skip reconnect" sticky.** Calling `mcp__plugin_nova_nova__authenticate` as a probe deletes any valid stored refresh token before starting a new flow — there's no read-only "am I authed?" call. Combined: a stale `plugin:nova:nova` entry in Claude Code's `mcp-needs-auth-cache.json` makes the host skip reconnection (logs show "Skipping connection (cached needs-auth)"); the orchestrator's instinct is to call `authenticate`, which wipes a refresh token that would otherwise have worked. **Rule:** before invoking Nova auth, check `mcp-needs-auth-cache.json` and delete the stale entry first; only call `authenticate` when a real tool call returns an auth error. (2026-05-05, from canopy observations 5e1816eab225, 9c2a16715e20, e05b11b195ca, b3fe2dc57466) **2026-05-21 update:** Nova plugin v1.1.0 ships PAT auth (no OAuth flow), so the destructive-`authenticate` and refresh-token-persistence vectors are moot on the PAT path. The `mcp-needs-auth-cache.json` delete-and-restart recipe still applies if the Nova entry ever lands in "Needs authentication" stuck state — per Braxton on voidcraft-labs/nova-plugin#11.
- **MCP server `PATH` is frozen at session start.** `ace-mobile-server` (and any MCP) is spawned once at session start; its `PATH` env is captured at that instant. Tools installed mid-session (Maestro via brew, etc.) are invisible — every call returns `spawn maestro ENOENT` even after symlinking into `node_modules/.bin`, because the server's actual cwd is unpredictable. The bootstrap skill has no mechanism to signal a restart of the MCP server. **Rule:** bootstrap skills that install new CLIs must instruct the user to fully restart Claude Code (not just `/reload-plugins` — that doesn't re-spawn MCP servers with a fresh PATH) before the new tool becomes visible. (2026-05-02, from canopy observations 7d9f142d22a5, 5d1972a329b8)
- **MCP create/mutate atoms need a verify-after-create read-back.** `connect_create_opportunity` and `connect_create_payment_units` returned a success response that the skill accepted as authoritative — but field divergence between the request and the stored values (amount, max_total, required_deliver_units, end_date) only surfaced in the post-hoc `-eval` rubric, after the bad state had shipped through two downstream phases. Same class as the OCS pipeline silent-block (errors hide behind layered endpoints). **Rule:** any MCP atom that mutates an external system should do one read-back `get_*` call and assert the requested fields round-tripped; pre-flight checks at the MCP boundary close the class better than per-skill post-hoc evals. (2026-05-05, from canopy observation 38ab961991ae)
- **`connect-opp-setup` can silently reuse a stale opp from a prior run with the wrong payment-unit config.** A prior run's opp (`2f1d2768`) was reused without comparing its payment-unit amount ($20) against the current PDD's working assumption ($150/shop). The stale config persisted silently through two phases. Run independence (per CLAUDE.md) is supposed to prevent this — but reuse by ID/slug shortcuts can still leak. **Rule:** when any phase looks up an existing Connect/OCS/Drive artifact by ID, it must verify the artifact's key parameters against the current run's PDD before using it; mismatch → recreate, don't adapt. Same shape as the precondition-restore-don't-classify principle in CLAUDE.md. (2026-05-05, from canopy observation b9488677d06a)
- **`commcare_download_ccz` returns a manifest-only "lite" CCZ by default — multimedia binaries are not included.** Verify steps that grep the CCZ for media files silently false-negated successful uploads until an `include_multimedia` flag was added in a follow-up PR. Atom defaults that drop binary content to save bytes are reasonable, but the atom's docstring and any verify-step skill must call this out explicitly. **Rule:** when an MCP atom has a "lite vs full" mode, the default name should encode it (`download_ccz_lite` vs `download_ccz_full`) or the docstring must lead with the boundary, so downstream verify code can't accidentally rely on the lite shape. (2026-05-06, from canopy observation 7856212b3760)
- **Parallel Conductor worktrees racing on the same bug is a recurring pattern — surfaces 4× in 4 days of observations.** Two (sometimes three) sessions independently diagnosed and fixed the same upstream issue, both shipped PRs, one had to be rebased / dropped. No mechanism currently signals "another session is already on this." Worktree-aware `/ship` (the `landing-report` skill) is the partial answer for open PRs, but doesn't cover "we're both about to open the same PR." Worth a `canopy:claim <issue>` lock or a pre-commit hook that checks open PRs touching the same files. (2026-05-02 to 2026-05-06, from canopy observations 128c077f20ce, 490ea39a4eda, c2932c83621d, fe04438e2d08)
- **Cross-MCP code duplication (extractCsrfToken in both OCS and Connect Playwright backends) silently diverges; the fix has to be applied twice.** When 0.13.19 added `domainFilter` support to `extractCsrfToken` in `mcp/connect/auth/playwright-session.ts` to fix a CSRF bug, the OCS copy at `mcp/ocs/auth/playwright-session.ts` was untouched and still had the unsafe pre-fix behavior — waiting to re-surface the same bug in a different backend. **Rule:** auth-layer helpers (`extractCsrfToken`, `isAuthenticated`, session-state load/save) should live in one shared module under `mcp/lib/auth/` rather than be copy-pasted per MCP; class-level preventer over per-MCP fixes. Audit other shared shapes (CSRF, login-page detection, cookie save/load) for the same drift class. (2026-05-06, from canopy observation fe70fe9827b1)
- **`artifact-manifest.ts` describes the canonical layout; actual Drive paths can drift from it without breaking ACE itself.** A summary reader built against `artifact-manifest.ts` returned all-null sections on first deploy because the plugin's actual write paths had drifted from the manifest (apps under `runs/<id>/2-commcare/`, Connect/OCS/Training at opp root not under `runs/`, etc.). The manifest tests passed because the test fixture was kept in lockstep with the manifest, not with the producing skills. **Rule:** any external reader (canopy summary, evals, dashboards) that consumes Drive artifacts must validate against a live opp's actual paths (`drive_list_folder` walk), not just trust `artifact-manifest.ts`; conversely, when a producing skill changes its output path, update the manifest AND the test fixture in the same PR. (2026-05-06, from canopy observation a585fbf8ac27)
