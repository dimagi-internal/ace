## Preferences

- **Runtime-health checks should be WARN, not FAIL.** When adding `/ace:doctor` (or similar) checks that test whether a user is ready to *use* a feature (vs. whether the install works), use WARN. A user doing a design-review-only run shouldn't see FAIL for missing OCS config. FAIL is reserved for "tool won't run at all." (2026-04-15, from P3 implementation)
- **Skills that read external-human inputs must fail loudly, not improvise.** If a skill's input is `producedBy: 'external'` in the artifact manifest and the file isn't there, the skill must stop with an actionable error — never invent content. The orchestrator is responsible for capturing the input; the skill is responsible for defending against the bypass path (`/ace:step`). (2026-04-15, from P1 implementation)
- **Smoke-test scripts on the current machine before committing.** Bash / shell changes to `bin/` that look right in the diff can still break on live env data (quoting, quotes, empty values, etc.). Run the script at least once locally with `--here` or equivalent before staging. Caught a quote-stripping bug on 2026-04-15 that would have shipped otherwise.
- **Passing tests can still hide fixture drift against a spec/manifest.** A green `npm test` is not the same as "the fixture matches the canonical shape declared in `lib/artifact-manifest.ts`." The 2026-04-16 scout found three gaps (stale `state.yaml`, missing `test-prompts.md`, missing Phase 4–6 artifacts) that all passed CI because the test was pinned to `upToPhase: 'connect'`. When a change to the manifest / phase structure / skill contracts lands, re-check fixtures against the full manifest — don't rely on the test alone. (2026-04-16)
- **3-fixture coverage model.** `CRISPR-Test-001` = partial input fixture for `ocs-agent-setup` (Phase 1–3 inputs only); `CRISPR-Test-002` = focus-group archetype stress fixture (Phase 1–3); `CRISPR-Test-003-Turmeric` = complete E2E fixture with every required artifact stubbed across all 6 phases. When adding new skills / artifacts, update CRISPR-Test-003-Turmeric first (it's the one the manifest test validates end-to-end). (2026-04-16)
- **qa/eval two-phase pattern is canonical.** A skill that exercises an artifact at runtime (chat with a deployed bot, facilitate a group session, capture audio) should split into `-qa` (capture evidence + structural checks, no LLM) and `-eval` (judge evidence with LLM-as-Judge). Every `-eval` writes a verdict YAML to `verdicts/<skill>-<mode>.yaml` using the uniform shape in `skills/README.md § QA vs Eval`. Skills that produce their own evidence inline use a `## LLM-as-Judge Rubric` self-eval section. The umbrella `opp-eval` skill aggregates every `verdicts/*.yaml` into a run-level scorecard with recommendations — add a rubric to a skill and it auto-registers. (2026-04-19, from qa-eval-iteration-loop cycle)
- **OCS can silently block `publishChatbotVersion` on pipeline errors.** A non-existent `collection_index_id` attached via `ocs_attach_knowledge` lands without error at the pipeline-patch layer (200 OK), but then every subsequent publish call is blocked with the opaque UI message *"Unable to create a new version when the pipeline has errors."* The draft ends up correctly configured, but the widget keeps serving v1 (empty post-clone) forever. Rule: pre-flight validate collection ids exist on the team before attaching — see `scripts/bootstrap-ocs-golden-template.ts` `listCollectionIndexIds` helper (scrapes the edit page since OCS has no REST endpoint for this). Same hardening needed in `ocs-agent-setup` for every per-opp bot clone. (2026-04-19, discovered when golden template scored 3.84/10 FAIL)
- **Archetype branching is a single-skill one-PR unit of work.** Adding a `## Archetypes` section with per-archetype branches to a SKILL.md is consistently ~1 file of substantive change + CHANGELOG + version bump, tests stay green, shippable autonomously in one cycle. Two shipped this way in 2026-04-19 (pdd-to-test-prompts, llo-invite). When auditing remaining skills for silent atomic-visit defaults, each can be its own small PR — no need to batch. (2026-04-19)
- **Defensive state init on bypass paths.** `/ace:run` initializes `state.yaml` as part of "Starting a New Opportunity," but operators bypass the orchestrator via `/ace:step <skill> <opp>` or by dispatching a phase agent directly. Every entry path that touches state must tolerate a missing `state.yaml` and init first. `/ace:step` owns this for its path (see `commands/step.md` step 4); phase-agent Agent-tool dispatches are expert paths and should at minimum not crash on missing state — skip the status update with a single-line warning. Motivation: I hit this myself in the cosmetics-fgd-pilot setup when I bypassed `/ace:run` and subsequent skill dispatches had no state file to update. (2026-04-19)
- **OCS team-scoping is enforced at publish, not attach.** `ocs_attach_knowledge` with a `collection_index_id` that doesn't exist on the current team silently succeeds at the pipeline-patch layer (returns `ok: true`). Every subsequent `publishChatbotVersion` is then blocked with the opaque UI message *"Unable to create a new version when the pipeline has errors."* The real error lives on the pipeline-save response body under `errors.node.<id>.<field>`, which `/versions/create` never calls. As of 0.5.1, `publishChatbotVersion` pre-flights through `/pipelines/data/` to surface this as a typed `PipelineValidationError` with node + field — the silent-block class is closed at the MCP bottleneck. Cross-team collection references (template on team A referencing a collection on team B) hit the same rejection path; OCS does not support them. (2026-04-20, verified via Path C experiment + Iter 8 clone)
- **Pipeline-save response has two error shapes.** The OCS pipeline-save endpoint returns errors in one of two shapes depending on what failed: `{ errors: ["string1", "string2"] }` for edge-level / graph-topology errors, or `{ errors: { node: { "<node-id>": { "<field>": "<msg>" } } } }` for node-level (parameter / reference) errors. Any code that parses this response must handle both shapes — checking only the top-level string array misses all node-level errors. `extractPipelineErrors` in `mcp/ocs/backends/pipeline-patch.ts` (0.5.1) is the canonical extractor; use it rather than re-implementing. (2026-04-20)
- **Metadata files (`~/.ace/*.json`, `connect-ocs-bot.json`, etc.) are hypotheses, not truths.** The 2026-04-09 snapshot of `~/.ace/connect-ocs-bot.json` listed `shared_collection_id: 718`, which was stale by 2026-04-19. Anchoring subagent reasoning on that id propagated a wrong premise across two cycles. For any metadata file older than ~7 days that's load-bearing for the current cycle, probe the relevant fact against live state before acting. When you correct stale metadata, update the file in the same cycle to prevent re-propagation. (2026-04-20, from collection-clone-and-mcp-preflight cycle)
- **Django forms can silently-accept with missing required fields.** `add_collection_files` accepts a POST without `chunk_size` / `chunk_overlap` and returns 302 (redirect to collection home), but produces zero chunks during indexing — retrieval then silently never works. Any MCP wrapper for a Django form-backed endpoint must verify it's sending every field the form declares as required; unit tests against mock request layers miss this. (2026-04-20, from uploadCollectionFiles fix in 0.5.1)
- **Prefer class-level preventers to instance-level checks.** The first-draft fix was "add a doctor check for ACE_DRIVE_ROOT_FOLDER_ID." The shipped fix was "diff the full `.env.tpl` key set against the installed `.env`." Second draft is ~15 lines of bash for ~unlimited future coverage; it also catches the shared-collection triple and every future var automatically. Same pattern as the 0.5.1 MCP pre-flight (one bottleneck closes the whole silent-block class). Rule: when proposing a check/validation, ask "what's the class of thing this detects?" and write the check at the class level, not the instance level, whenever the class boundary is cheap to express. (2026-04-20, from env-drift-adoption-blockers cycle)
- **For adoption-blockers lens, don't trust the tool under audit as its own oracle.** Doctor reported `STATUS: COMPLETE` while the installed `.env` was missing 8 keys from `.env.tpl` — doctor only validated 3 of 16 keys. The scout almost moved on based on that green light. Read primary sources (the `.env` file, `.env.tpl`, user-facing docs) directly before trusting any "everything is green" summary from the same tooling that may itself be the gap. (2026-04-20, from env-drift-adoption-blockers cycle)
- **`.env.tpl` drifts forward across releases; installed `.env` files don't auto-update.** The canonical fix for users is `op inject -i .env.tpl -o <env-path> --account dimagi.1password.com`, which preserves existing 1Password-backed values while picking up new keys. As of 0.5.4, `/ace:doctor`'s new `env_drift` check surfaces which keys are missing from the installed `.env` with the exact `op inject` command as the fix hint. When `.env.tpl` gains a new var, no doctor-code change is needed — the diff picks it up automatically. (2026-04-20, from env-drift-adoption-blockers cycle)
- **Sibling-checkout merge discipline: verify `git branch --show-current` before any `pull/merge/push` chain.** The main checkout at `~/emdash-projects/ace/` can be left on a stale feature branch from previous work (e.g. `feat/run-smart-defaults` after PR #41 was merged and its remote deleted). A naive `cd ~/emdash-projects/ace && git pull --rebase && git merge <branch> && git push` lands the merge on that stale branch and can revive the remote branch with a fresh push. Canonical form: `cd ~/emdash-projects/ace && [ "$(git branch --show-current)" = "main" ] || git checkout main && git pull --ff-only && git merge <branch> --no-ff && git push`. Recovery if this happens: `git checkout main && git merge --ff-only <accidental-branch> && git push origin main` (works when the accidental merge commit has the real main tip as an ancestor), then delete the revived remote branch. (2026-04-20, from env-drift-adoption-blockers cycle)
