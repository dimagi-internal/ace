## Meta-observations

1. **Doc-vs-frontmatter drift is a recurring class on this surface.** Six off-by-one phase descriptions all drift in the same direction (+1) and presumably from the same source — a pre-renumbering era when an earlier "phase 0" or pre-PDD phase existed. Without a CI assertion the drift survived multiple releases unnoticed. Generalizes: any time numbered ordinals appear in two places (frontmatter + heading + description), a structural test is cheap and catches the class.

2. **Procedure-vs-reference split needs an editorial discipline, not just a structural one.** The 2026-05-10 perf-lens session correctly split the orchestrator doc, but ~2 weeks later the procedure side has accreted back ~166 lines of `**Why:**` paragraphs. The split is structurally correct; the day-to-day editing convention is "where does this new content go?" and the default answer keeps landing in the procedure doc because the procedure is what new sessions read first. Without a per-PR editor checklist ("if you added a `**Why:**` paragraph in procedure, did you mean to add it to reference?"), the split silently degrades.

3. **0.13.116 gate-brief removal was a half-finished sweep.** Reference doc cleanly explains the removal; producer-side instructions to create/read gate-briefs survived in 5 places. Pattern: a class-level rename/removal needs both a "tell the reader why" pass (done well) AND a "delete every instruction to do the old thing" pass (skipped). The CI gate in Proposal 3 catches future instances of this class.

4. **Validator type-only checks let documented enums rot.** `verdict` enum drift in `closeout.md` (`closed` not in the documented `pass | proceed | …` list) is invisible because `lib/run-state-validator.ts` only asserts string type, not enum membership. Either the enum should be enforced (would fail today, exposing the drift) or the docs should drop the enum claim. Same pattern as the MCP-vs-skill-doc drift learnings — if you document an interface contract, the implementation enforces it or the contract drifts.

5. **The scout almost re-proposed the 2026-05-10 orchestrator structural split.** Reading the procedure-vs-reference duplication, my first instinct was "this needs another structural split." Checking the prior run log corrected the framing: the split already happened; the problem is editorial discipline, not architecture. Lesson reinforced: always read prior runs before proposing structural changes. The skill enforces this via "Read `learnings.md` before doing anything else" but the prior runs/ folder is equally load-bearing.

6. **Bundle-or-split is per-class, not per-prior-PR.** Prior session shipped 11 separate PRs; this session bundles two classes (off-by-one phase + stale gate-brief) into one PR. The two classes share repo surface (`agents/*.md`), share validation surface (one CI test in Proposal 3 covers both), and one prerequisite (test in Proposal 3 needs Proposal 1's content fixed first). Bundling them here matches the staged-PR principle's intent (small PRs, each independently mergeable) better than shipping 11 micro-PRs for 11 individual lines.
